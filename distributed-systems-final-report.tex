\begin{abstract}
	Hermes is a distributed real-time messaging system designed as a feasible prototype for a university project in distributed systems. The goal of the project is not to replicate a full-scale commercial messaging platform, but to demonstrate the core principles of distributed communication through a compact yet meaningful application. The system supports authenticated one-to-one text messaging, allowing users to register, log in, and exchange messages through a desktop client. 
	
	The architecture is based on a client-server model with a distributed backend. Users authenticate through HTTP/REST, receive a JWT token, and then use a persistent WebSocket connection for real-time message exchange. Messages are handled by an event-driven backend built around Kafka, while Redis is used to support presence and routing information and PostgreSQL stores user credentials. On the client side, SQLite is used to persist local message history and synchronization metadata, enabling offline access and later reconciliation after disconnections.
	
	The project focuses on key distributed-systems concepts such as scalability, replication, fault tolerance, recovery, and eventual consistency. In particular, the system prioritizes availability over strict consistency and uses logical ordering mechanisms to preserve a meaningful message order without requiring a globally synchronized state. Hermes therefore provides a concrete example of how distributed components can cooperate to deliver a responsive, resilient, and extensible messaging service within a realistic development scope.
\end{abstract}

\section{Concept}\label{concept}

\begin{itemize}
	\item \textbf{Type of product:} The project is a real-time messaging service composed of a distributed backend and a desktop client application with a graphical user interface. The backend exposes web services and supports message exchange through a client-server architecture.
	
	\item \textbf{Use case description:} The software enables one-to-one chat between users:
	\begin{itemize}
		\item \emph{What:} Users can register and log in, then exchange instant text messages. Each message sent by a user is delivered to the intended recipient if they are online, or synchronized later if they are temporarily offline. The system does not maintain a permanent server-side conversation history; instead, message history is stored locally on the client, which also allows users to access previously received messages and compose messages while disconnected from the network.
		\item \emph{Where:} Users interact with the system from any location with an Internet connection, typically using personal devices such as laptops or desktop computers.
		\item \emph{When and how frequently:} Interaction is on-demand and frequent: users may send and receive messages continuously during a chat session. The platform supports both sporadic use and sustained conversation flow.
		\item \emph{How and which devices:} Users interact with the system through a desktop client with a graphical user interface. The client first connects to the backend through HTTP/REST to perform authentication and initial synchronization, obtaining an access token and retrieving messages received since the last access. Once this phase is completed, the client opens a persistent WebSocket connection for real-time updates.
		\item \emph{Data storage:} The system stores user-related data and message-related data. User accounts are stored in a relational database (PostgreSQL), while the messaging backbone relies on Kafka to support the exchange of messages within the distributed backend. On the client side, messages and synchronization data are stored locally in SQLite.
		\item \emph{Roles:} The main role is the end user, i.e., a chat participant. A system administrator role may exist at infrastructure level, but it is not central to the scope of this project. Standard user actions are symmetric between users.
	\end{itemize}
	
	\item \textbf{Why distribution is needed:} The system adopts a distributed architecture to satisfy the main requirements of a modern real-time messaging platform. In particular, distribution allows the system to scale more easily by adding service instances and expanding the supporting infrastructure when needed. It also improves fault tolerance and availability, since the failure of a single component does not necessarily compromise the whole service. Replication of internal components helps avoid single points of failure and supports durability of important state, while the event-driven nature of the system makes message handling more asynchronous and loosely coupled. Distribution is also useful for recovery, because it allows the system to resume operation after transient failures and to reconcile state once connectivity is restored. More generally, this architectural choice improves modularity and separation of concerns, since different parts of the system can be responsible for distinct tasks such as authentication, message propagation, presence management, and persistence. Following the CAP theorem, the system gives priority to \emph{Availability} rather than strict \emph{Consistency}. For this reason, it adopts an \emph{eventual consistency} approach: messages are accepted and propagated as soon as possible, and a consistent state is recovered later through synchronization mechanisms. To support this behavior, the system can use Lamport clocks to preserve a meaningful logical ordering of events without requiring a fully synchronized global state. This choice is appropriate for a chat application, where responsiveness is more important than enforcing strong consistency at every instant.
\end{itemize}

\section{Requirements Elicitation and Analysis}\label{requirements}

\subsection{Functional Requirements}

\begin{itemize}
	\item \textbf{FR1 -- User registration and authentication:} The system shall allow a user to create an account and authenticate securely before accessing the messaging service.
	
	\textbf{Acceptance criterion:} A user can register with valid credentials, log in successfully, and access the application only after authentication.
	
	\item \textbf{FR2 -- One-to-one messaging:} The system shall allow authenticated users to exchange private text messages with other registered users.
	
	\textbf{Acceptance criterion:} A user can send a text message to another registered user, and the recipient can read it if they are connected or later after synchronization.
	
	\item \textbf{FR3 -- Real-time message delivery:} The system shall deliver messages in real time when both users are online.
	
	\textbf{Acceptance criterion:} When two users are connected, a message sent by one user becomes visible to the recipient without requiring manual refresh.
	
	\item \textbf{FR4 -- Offline message synchronization:} The system shall allow users to recover messages that were received while they were temporarily offline.
	
	\textbf{Acceptance criterion:} After a user reconnects and logs in again, previously missed messages are retrieved and displayed in the client.
	
	\item \textbf{FR5 -- Local conversation storage:} The system shall keep a local history of messages on the client side so that users can access past conversations even without an active network connection.
	
	\textbf{Acceptance criterion:} A user can open the client and view previously stored messages from the local database.
	
	\item \textbf{FR6 -- Message ordering support:} The system shall preserve a meaningful order of messages within a conversation, even when delivery is asynchronous.
	
	\textbf{Acceptance criterion:} Messages are displayed in an order that is consistent with the logical progression of the conversation.
\end{itemize}

\subsection{Non-Functional Requirements}

\begin{itemize}
	\item \textbf{NFR1 -- Availability:} The system shall remain usable even when some internal components are temporarily unavailable.
	
	\textbf{Acceptance criterion:} A temporary failure of one internal component does not completely stop the service, and the system resumes normal operation after recovery.
	
	\item \textbf{NFR2 -- Eventual consistency:} The system shall ensure that the state observed by users converges over time after temporary disconnections or delays.
	
	\textbf{Acceptance criterion:} After synchronization, users eventually observe a consistent view of exchanged messages.
	
	\item \textbf{NFR3 -- Responsiveness:} The system shall support low-latency interaction suitable for a real-time chat application.
	
	\textbf{Acceptance criterion:} Sending and receiving a message should feel immediate from the user's perspective during normal operation.
	
	\item \textbf{NFR4 -- Reliability and recovery:} The system shall recover gracefully from transient failures and support state reconciliation after disconnections.
	
	\textbf{Acceptance criterion:} After a disconnection, the system is able to restore missing messages and reestablish a consistent conversation state.
	
	\item \textbf{NFR5 -- Scalability:} The system shall support growth in the number of users, messages, and concurrent connections without requiring a redesign of the overall architecture.
	
	\textbf{Acceptance criterion:} The system can be extended by adding more service instances or infrastructure resources while preserving the same logical behavior.
	
	\item \textbf{NFR6 -- Durability:} The system shall ensure that messages are not lost due to transient failures.
	
	\textbf{Acceptance criterion:} Messages sent by a user are eventually delivered or recoverable after temporary failures.
	
	\item \textbf{NFR7 -- Maintainability:} The system shall be structured in a modular way so that different responsibilities can evolve independently.
	
	\textbf{Acceptance criterion:} Changes to one subsystem, such as authentication or message delivery, can be introduced with limited impact on the others.
\end{itemize}

\subsection{Implementation Requirements}

\begin{itemize}
	\item \textbf{IR1 -- Limited development scope:} The system shall be designed and implemented by a single developer within a limited time budget (approximately 90 hours).
	
	\textbf{Acceptance criterion:} The final system includes a working prototype that covers the core functionalities within the given time constraints.
	
	\item \textbf{IR2 -- Local deployment environment:} The system shall be deployable and testable on a single machine using a local environment.
	
	\textbf{Acceptance criterion:} The system can be executed and tested locally without requiring external infrastructure.
	
	\item \textbf{IR3 -- Use of container orchestration:} The system shall be deployable in a containerized environment using Kubernetes for educational purposes.
	
	\textbf{Acceptance criterion:} The system can be deployed and run using a local Kubernetes setup (e.g., Minikube).
	
	\item \textbf{IR4 -- Technology familiarity constraint:} The implementation should rely on technologies that are compatible with the developer’s existing knowledge and learning goals.
	
	\textbf{Acceptance criterion:} The chosen technologies allow the system to be implemented within the project constraints and support the intended learning objectives.
\end{itemize}

\subsection{Glossary}

\begin{itemize}
	\item \textbf{JWT (JSON Web Token):} A token used to represent the authenticated identity of a user and to authorize requests to the system.
	\item \textbf{REST (Representational State Transfer):} An architectural style for designing networked applications, typically based on HTTP, used here for client-server communication during authentication and synchronization.
	\item \textbf{WebSocket:} A communication protocol that enables persistent bidirectional interaction between the client and the server for real-time message exchange.
	\item \textbf{Kafka:} A distributed event streaming platform used as the backbone of the system to propagate messages between components.
	\item \textbf{Event-driven architecture:} A design paradigm in which system components communicate through the production and consumption of events, enabling loose coupling and asynchronous interaction.
	\item \textbf{Lamport clock:} A logical clock mechanism used to maintain a consistent ordering of events in a distributed system without relying on synchronized physical time.
	\item \textbf{Eventual consistency:} A consistency model in which the system guarantees that all participants will converge to the same state over time, even if temporary inconsistencies occur.
	\item \textbf{Presence information:} Information that indicates whether a user is currently online and reachable, used to support message routing and delivery decisions.
	\item \textbf{Reconciliation:} The process of restoring a consistent system state after temporary inconsistencies or disconnections.
\end{itemize}
